# Lステップ管理画面 自動化ツール 要件定義書（改訂版）

> 最終更新: 2026-03-17

---

## 1. プロジェクト概要

### 目的

LSTEPの依存関係を把握し、ユーザーが新機能追加・設定変更を指示した際に、依存関係の影響を踏まえてミスなく実行できる半自動化環境を構築する。

**背景:**
現在、LSTEP内の依存関係が複雑で新機能を追加するときにミスが発生しやすい。依存関係を先に可視化・把握してから操作することで、安全に機能追加できるようにしたい。

### 技術方針
- Chrome DevTools Protocol（CDP）経由でログイン済みブラウザに接続
- Playwrightでブラウザ操作を実行
- CAPTCHA・ログイン認証は手動で事前に通しておく
- 操作間隔は2〜3秒を確保し、サーバ負荷を人間操作レベルに抑える

### 対象規模
- 1アカウント運用
- シナリオ配信：数十本
- タグ：数十個
- 友だち情報欄：数十項目
- アクション（テンプレート・リッチメニュー等）：数十個

## 2. 機能一覧

| # | 機能名 | 優先度 | 概要 | 役割 |
|---|--------|--------|------|------|
| F1 | シナリオ配信の設定取得・複製 | 高 | 既存シナリオの設定内容を取得し、複製・編集する | データ収集 + 書き込み |
| F2 | タグ設定の一覧取得・一括操作 | 高 | タグの一覧取得、新規作成、名称変更 | データ収集 + 書き込み |
| F3 | 友だち情報管理の設定取得・操作 | 高 | 友だち情報欄の一覧取得、追加、編集 | データ収集 + 書き込み |
| F4 | アクション依存関係マッピング | **コア** | アクション間の依存関係（トリガー→アクション→遷移先）を解析 | **プロジェクトの核心** |
| F5 | 構成ドキュメント自動生成 | **コア** | F4の結果をもとに、アカウント全体の構成ドキュメントをMarkdownで出力 | **「ミスなく実行」の判断基盤** |

> **設計思想:** F1〜F3は「F4のためのデータ収集手段」。F4・F5がこのプロジェクトの核心機能。

## 3. 共通基盤

### 3.1 CDP接続モジュール

```
機能: ログイン済みChromeブラウザへのCDP接続確立
入力: CDPエンドポイント（デフォルト: http://localhost:9222）
出力: Playwright Browserインスタンス
処理:
  1. chromium.connectOverCDP() でブラウザに接続
  2. 既存コンテキスト・ページを取得
  3. Lステップ管理画面のタブを特定（URLに linestep.net を含むもの）
  4. タブが無い場合は新規タブでLステップを開く
```

### 3.2 操作制御（レートリミット）

```
方針:
  - 全てのページ遷移・クリック操作の後に 2〜3秒のランダムウェイト
  - ページ読み込み完了を waitForLoadState('networkidle') で確認してから次の操作
  - エラー発生時は3回までリトライ（リトライ間隔: 5秒）
```

### 3.3 データ保存形式

```
保存先:
  data/          # 取得データ（JSON生データ）
  data/backups/  # 書き込み前の自動バックアップ
  docs/          # 生成ドキュメント（Markdown）

形式: JSON（生データ）、Markdown（ドキュメント出力）
ファイル命名: {機能名}_{YYYYMMDD_HHmmss}.{拡張子}
```

### 3.4 Dry-runモジュール（新規）

```
機能: 全書き込み操作の事前シミュレーションと保護

Dry-runフロー:
  1. 操作計画を生成（何を・どの順で・何に影響するか）
  2. F4の依存関係データが存在する場合、影響範囲を確認・表示
  3. ユーザーに提示して確認を取る
  4. 承認後、本実行前に対象データをJSONバックアップ（data/backups/）
  5. 実行中にエラーが発生した場合は自動ロールバック

確認プロンプト出力例:
  [DRY RUN] 以下の操作を実行します:
    ① シナリオ「初回フォロー」を複製 → 「初回フォロー_コピー」
    ② タグ「新規」を付与条件に追加

  依存関係への影響:
    ⚠ タグ「新規」はシナリオB・Cでも参照されています
    ✓ リッチメニューへの影響なし

  実行しますか？ (Y/n):

制約:
  - F1-3, F2-2, F3-2 の全書き込み操作はDry-run確認後でないと実行不可
  - バックアップなしの書き込みは禁止
```

### 3.5 依存チェックゲート（新規）

```
機能: 書き込み操作実行前にF4の依存関係データを参照し、影響範囲を警告する

処理:
  1. F4の依存関係JSONが存在するか確認
  2. 変更対象（シナリオ名/タグ名/情報欄名）を依存グラフで検索
  3. 影響するノード（他のシナリオ・タグ・リッチメニュー等）を列挙
  4. 影響がある場合 → 警告を表示してユーザーに判断を委ねる
  5. 影響なし → そのまま実行

これが「ユーザーが指示したらミスなく実行」の核心機能。
```

## 4. 各機能の詳細

### F1: シナリオ配信の設定取得・複製

#### F1-1: シナリオ一覧取得
```
処理フロー:
  1. シナリオ配信の管理ページに遷移
  2. シナリオ一覧テーブルからデータを抽出
     - シナリオ名
     - ステータス（配信中/停止中/下書き）
     - 配信数
     - 作成日・更新日
  3. JSON形式で保存

出力例:
{
  "scenarios": [
    {
      "id": "xxx",
      "name": "初回来院後フォローアップ",
      "status": "配信中",
      "send_count": 150,
      "steps": [...]
    }
  ]
}
```

#### F1-2: シナリオ詳細取得
```
処理フロー:
  1. 各シナリオの編集画面に遷移
  2. ステップ構成を抽出
     - 各ステップの配信タイミング（即時/○日後/○時間後）
     - メッセージ内容（テキスト/画像/リッチメッセージ等）
     - 分岐条件（タグ有無、友だち情報の値など）
     - アクション設定（タグ付与/リッチメニュー切替など）
  3. JSON形式で保存
```

#### F1-3: シナリオ複製（書き込み・Dry-run必須）
```
処理フロー:
  1. Dry-runモジュールで操作計画を提示・確認
  2. 依存チェックゲートで影響範囲を確認
  3. 対象データをバックアップ
  4. 複製元シナリオを指定
  5. Lステップ標準の複製機能があればそれを使用
  6. 複製後、指定された項目（シナリオ名、配信タイミング等）を編集
  7. 下書き状態で保存
```

---

### F2: タグ設定の一覧取得・一括操作

#### F2-1: タグ一覧取得
```
処理フロー:
  1. タグ管理ページに遷移
  2. 全タグの情報を抽出
     - タグ名
     - タグフォルダ（あれば）
     - 該当友だち数
  3. JSON形式で保存
```

#### F2-2: タグ一括作成（書き込み・Dry-run必須）
```
処理フロー:
  1. Dry-runモジュールで操作計画を提示・確認
  2. 作成するタグのリスト（JSON/配列）を入力として受け取る
  3. タグ管理ページで1つずつ作成
     - 新規作成ボタン → タグ名入力 → 保存
     - 操作間隔を確保
  4. 作成結果をログ出力
```

---

### F3: 友だち情報管理の設定取得・操作

#### F3-1: 友だち情報欄の一覧取得
```
処理フロー:
  1. 友だち情報管理ページに遷移
  2. 全情報欄の設定を抽出
     - 項目名
     - 型（テキスト/選択肢/日付など）
     - 選択肢の値（選択肢型の場合）
     - フォルダ分類
  3. JSON形式で保存
```

#### F3-2: 友だち情報欄の追加・編集（書き込み・Dry-run必須）
```
処理フロー:
  1. Dry-runモジュールで操作計画を提示・確認
  2. 依存チェックゲートで影響範囲を確認
  3. 対象データをバックアップ
  4. 追加/編集する情報欄の定義をJSON入力で受け取る
  5. 管理画面で操作を実行
  6. 結果をログ出力
```

---

### F4: アクション依存関係マッピング（コア機能）

#### 目的
「タグAが付与される → シナリオBが発動 → リッチメニューCに切替」のような連鎖を可視化する。
新機能追加時の影響範囲を事前に把握し、ミスを防ぐ。

#### F4-1: 全アクション設定のスキャン
```
処理フロー:
  1. 以下の各設定画面を巡回してアクション設定を収集
     - シナリオ配信の各ステップ
     - テンプレート
     - リッチメニュー
     - 自動応答
     - 友だち追加時設定
     - カスタム検索管理
     - 予約管理（あれば）
  2. 各アクションから以下を抽出
     - トリガー条件（何がきっかけで発動するか）
     - 実行アクション（タグ付与/除外、シナリオ開始/停止、リッチメニュー切替、テンプレート送信、友だち情報更新 等）
     - 遷移先（次にどの設定に影響するか）
  3. 全データをJSON形式で構造化して保存
```

#### F4-2: 依存関係グラフ構築
```
処理フロー:
  1. F4-1のデータからノード（各設定要素）とエッジ（依存関係）を抽出
  2. 依存関係を以下の観点で分類
     - 順方向: A → B（Aが発動するとBに影響）
     - 逆方向: Bが機能するにはAが前提
     - 循環: 相互依存がないかチェック
  3. Mermaid記法のフローチャートとして出力

出力例:
  graph TD
    友だち追加 --> タグ:新規
    タグ:新規 --> シナリオ:初回フォロー
    シナリオ:初回フォロー --> リッチメニュー:通常メニュー
    シナリオ:初回フォロー --> タグ:フォロー済
```

---

### F5: 構成ドキュメント自動生成（コア機能）

#### 目的
依存関係を人間が読める形で出力し、新機能追加時の意思決定・指示の精度を上げる。

#### 出力内容
```
Markdownファイルとして以下を含む構成ドキュメントを生成:

1. アカウント構成サマリー
   - シナリオ数、タグ数、友だち情報欄数、テンプレート数

2. シナリオ一覧と各ステップの概要
   - 各シナリオのステップ数、配信タイミング、分岐条件

3. タグ一覧と用途
   - 各タグがどのシナリオ・アクションで使われているか

4. 友だち情報欄一覧と用途
   - 各情報欄がどこで参照・更新されているか

5. アクション依存関係図（Mermaid）
   - 全体フロー図
   - シナリオ単位の詳細フロー図

6. 注意事項・潜在的問題点
   - 使われていないタグ・情報欄
   - 循環依存
   - 孤立しているシナリオ（トリガーが不明）
```

## 5. 実行環境・セットアップ

### 前提条件
- macOS（M4 Mac）
- Node.js インストール済み
- Playwright インストール済み（`npm install playwright`）
- Claude Code 利用可能

### Chrome起動手順（毎回）
```bash
# 1. Chromeを全て閉じる
# 2. デバッグポート付きで起動
/Applications/Google\ Chrome.app/Contents/MacOS/Google\ Chrome \
  --remote-debugging-port=9222

# 3. Lステップにログイン（手動）
# 4. Claude Codeからスクリプトを実行
```

### ディレクトリ構成（想定）
```
lstep-automation/
├── src/
│   ├── connection.js      # CDP接続モジュール
│   ├── utils.js           # 共通ユーティリティ（ウェイト、リトライ等）
│   ├── dry-run.js         # Dry-runモジュール（新規）
│   ├── dependency-gate.js # 依存チェックゲート（新規）
│   ├── scenarios.js       # F1: シナリオ関連
│   ├── tags.js            # F2: タグ関連
│   ├── friend-fields.js   # F3: 友だち情報関連
│   ├── dependencies.js    # F4: 依存関係マッピング
│   └── doc-generator.js   # F5: ドキュメント生成
├── data/
│   └── backups/           # 書き込み前の自動バックアップ
├── docs/                  # 生成ドキュメント保存先
├── selectors/
│   └── selectors.json     # セレクタ外部設定ファイル
├── package.json
└── README.md
```

## 6. 制約事項・リスク

| 項目 | 内容 |
|------|------|
| ログイン維持 | セッション切れ時は手動で再ログインが必要 |
| UI変更リスク | Lステップの画面更新でセレクタが変わる可能性あり。セレクタは `selectors/selectors.json` に切り出して保守性を確保 |
| 操作速度 | 人間の操作速度と同程度に制限（2〜3秒間隔） |
| 利用規約 | サーバ負荷をかけない範囲で利用。大量一括操作は避ける |
| **Dry-run必須** | F1-3, F2-2, F3-2 の全書き込み操作はDry-run確認後でないと実行不可 |
| **バックアップ必須** | 書き込み前に対象データをJSONでバックアップ。保存先: `data/backups/` |
| **依存チェック必須** | F4のデータが存在する場合、書き込み前に影響範囲を必ず表示 |
| 画面構造の事前調査 | 実装前にLステップ管理画面のDOM構造を調査する必要あり |

## 7. 開発フェーズ

### Phase 1: 基盤構築 + F1検証（MVP）
- CDP接続モジュール
- 操作制御（ウェイト・リトライ）
- **Dry-runモジュール**（新規）
- F1-1: シナリオ一覧取得
- F1-2: シナリオ詳細取得
- → 動作確認・セレクタ安定性を検証してから次Phaseへ

### Phase 2: 全データ収集層の完成
- F2-1: タグ一覧取得
- F3-1: 友だち情報欄一覧取得

### Phase 3: 依存関係マッピング（コアゴール）
- F4-1: 全アクション設定のスキャン
- F4-2: 依存関係グラフ構築（Mermaid出力）
- F5: 構成ドキュメント自動生成
- **依存チェックゲート**（新規）

### Phase 4: 書き込み系 + 安全な操作実行
- F1-3: シナリオ複製（Dry-run + ロールバック付き）
- F2-2: タグ一括作成（Dry-run + ロールバック付き）
- F3-2: 友だち情報欄の追加・編集（Dry-run + ロールバック付き）
